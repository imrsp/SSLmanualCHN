# 术语规则

## 术语来源

本项目的术语体系分两层：

- `docs/glossary.csv`：唯一的结构化术语源，供脚本和 Agent 使用。
- 本文档：解释术语策略、适用边界、允许保留英文的情形，以及如何维护 glossary。

如果两者冲突，以 `docs/glossary.csv` 为准；然后应尽快把本文档补齐。

## glossary.csv 的职责

`docs/glossary.csv` 至少承担这些作用：

- 为翻译提供推荐译法。
- 为术语审计提供可机器读取的 `term_en` / `term_zh` / `note` / `context` 信息。
- 为后续人工校对提供统一参照。

变更术语时，不要只改报告、只改正文，或只改本文档。必须先改 `docs/glossary.csv`，再同步相关说明。

## 默认术语原则

- 产品名、协议名、软件名、品牌名通常保留英文。
- 界面按钮、菜单、页签、参数名通常保留英文原文。
- 可一般化的功能概念、说明性标题、操作动作，优先使用统一中文译法。
- 同一个英文概念不要在不同章节随意换译。
- 如果需要多个可接受译法，写入 `term_zh` 时用 `/` 分隔，并在 `note` 中说明条件。

## 允许保留英文的情形

以下情况可以不强制替换成中文：

- 该词本身就是界面标签、按钮名或菜单路径的一部分。
- 该词是行业中固定使用的英文名，改译反而降低可检索性。
- `docs/glossary.csv` 的 `note` 明确写了“保留英文”或等价说明。

当前术语审计还内置了少量默认允许保留英文的词：

- `VCA`
- `Aux`
- `Stem`
- `Dante Primary`
- `Dante Secondary`

如果某词也应属于此类，优先在 `docs/glossary.csv` 的 `note` 中写明，而不是在脚本里继续堆硬编码。

## 术语审计实际检查什么

`npm run audit:terminology` 当前的检查方式是：

1. 读取 `docs/glossary.csv`。
2. 读取 `content/en/manifest.json`，确定章节范围。
3. 扫描 `content/zh/pages/` 中已存在的译文文件。
4. 对每个 glossary 条目统计：
   - 英文术语在译文全文中出现了多少次。
   - 推荐中文译法在译文全文中出现了多少次。
5. 如果某术语在译文里保留了英文，但全文一次推荐中文都没有出现，且该词不属于“允许保留英文”，就写入 `reports/TERMINOLOGY_AUDIT.md`。

这意味着它是“覆盖审计”，不是语义理解器。它能发现明显未落地的术语，但不会自动判断每一次英文保留是否都错误。

## 术语审计的参考来源

`TERMINOLOGY_AUDIT` 当前直接引用这三处作为输入：

- `docs/glossary.csv`
- `content/en/manifest.json`
- `content/zh/pages`

报告本身也会把这三项写入“参考来源”章节。

## 当前边界

术语审计适合发现：

- glossary 中已有推荐译法，但译文整体仍只保留英文。
- glossary 条目重复，导致规则不可信。
- glossary 文件格式损坏。

术语审计不负责自动判定：

- 某个英文词在具体语境下是否应该保留英文界面名。
- 中文译法是否足够自然。
- 同义近义表达里哪一个更适合当前段落。

这些仍然需要人工审校。

## 维护规则

新增或修改术语时：

1. 先改 `docs/glossary.csv`。
2. 必要时更新本文档中的规则说明。
3. 运行 `npm run audit:terminology`。
4. 查看 `reports/TERMINOLOGY_AUDIT.md` 是否出现新的覆盖问题或重复条目。

不要在脚本里增加“禁用旧译法”之类的临时硬编码，除非明确要建立新的、长期维护的规则层。
